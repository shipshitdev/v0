# Session Summary - 2026-04-24

Extended `@shipshitdev/v0` into a broader Bun-first workspace scaffold and aligned it with `@shipshitdev/ui@0.6.0`.

## CLI Flow

Interactive flow now runs in this order:

1. Detect installed scaffold agents (`codex`, `claude`) and optional `gh`.
2. Ask for scaffold agent first.
3. Ask for project scope.
4. Ask for app surfaces with selector-style prompts.
5. Ask for default routes.
6. Ask whether to create a GitHub repo with `gh` (default: no).
7. Render a granular progress board with `pending`, spinner `loading`, `done`, and `failed` states.
8. Scaffold files.
9. Optionally run the selected agent.
10. Optionally create the GitHub repo and issue.
11. Run `bun install`.
12. Print per-app Bun scripts.
13. Start `apps/web` automatically when selected, unless disabled.

## Generated Workspace

Generated projects now include:

- `apps/web` - Next.js marketing surface
- `apps/app` - Next.js product surface
- `apps/desktop` - Electron desktop surface
- `apps/mobile` - Expo mobile surface
- `apps/extension` - Plasmo browser extension surface
- `apps/cli` - Commander-based CLI surface

Generated repo support files now include:

- `.agents/skills/`
- `.agents/memory/`
- `.claude` symlinked to `.agents`
- `.codex` symlinked to `.agents`
- selection-based repo workflow skills

Generated root scripts include:

- `dev:<app>` for each selected surface
- `deps:update`
- `postinstall` link repair for Bun isolated installs when `@shipshitdev/ui` lands under `.bun/`

## Shared UI Alignment

- The scaffold now emits `@shipshitdev/ui@^0.6.0`.
- Generated projects continue to import generic UI from the shared root package.
- The scaffold does not hardcode any old `0.5.x` shared UI pin anymore.

## Important Fixes

- GitHub setup remains opt-in and defaults to no.
- Progress steps are split more granularly so scaffold phases are visible.
- Bun remains the default install/runtime path throughout the generated workspace.
- The generated workspace still reserves `packages/` so downstream `rg ... apps packages` commands do not fail.
